# システム仕様

## エンドポイント

`LlmEndpoint` は `ITranslateEndpoint` を実装し、XUnity.AutoTranslatorから受け取った単一テキストを内部キューへ保持します。XUnity本体には変更を加えません。

## バッチ処理

通常モデルでは、同じ翻訳元・翻訳先言語の項目を最大 `MaxBatchSize` 件まで候補にします。候補は文字列単位で扱い、途中分割しません。バッチには最低1件を含めます。

1件目はサイズ推定にかかわらず追加します。2件目以降は、候補を追加した状態のsystemメッセージとuserメッセージ、プロバイダ用余白、想定出力を組み立て、そのUTF-8バイト上限推定が `MaxRequestTokens` 以下の場合だけ追加します。

`MaxBatchCharacters` のような文字数上限は設けません。日本語や中国語を過小評価しないよう、事前tokenize APIがないOllamaではUTF-8バイト数を安全側のトークン上限として使います。単独項目が推定上限を超えても分割や事前拒否はせず、最低1件として送信します。

8 GB GPUを基準とする初期値は、推定リクエスト予算8192、最大8件、実HTTP並列1です。`MaxRequestTokens` はLLMのcontext設定値ではなく、プラグイン内部のバッチ選択だけに使います。Ollamaの `num_ctx` を含むプロバイダのcontext設定は送信しません。設定値は実際のモデルまたはサーバーのcontext上限以下にする必要があります。Ollama側で実contextを増やすとKV cacheのVRAM消費が増える場合がありますが、このプラグインの `MaxRequestTokens` 自体はVRAMを割り当てません。

バッチ待機の初期値は300msです。同時表示UIだけでなく、100～300ms程度の間隔で順次キューへ入るメッセージも同じ要求へまとめ、秒単位のローカルLLM呼び出し回数を減らします。最大件数へ達した場合は300msを待たずに確定します。実リクエスト中に次の項目が蓄積し、その先頭項目の300msが経過済みなら、実行枠が空いた時点で追加待機せず送信します。

翻訳元・翻訳先言語は `ITranslationContext.SourceLanguage` と `ITranslationContext.DestinationLanguage` を直接使用します。エンドポイント固有の言語上書きは行いません。

## モデルプロファイル

通常モデルは、XMLの `<request>`/`<response>` 形式でバッチ翻訳するデフォルトプロファイルを使います。小型ローカルモデル（12B程度でも）が崩しにくい平易なXMLを用い、応答は正規表現で解析します。各項目はIDではなく元テキストの一致で対応付けるため、順序入れ替えや一部欠落があっても整合します。モデル名に `hy-mt2` が含まれる場合は専用プロファイルを選択し、1件ずつ処理します。モデル固有（デフォルト以外）の処理は `TranslationProfile` の派生クラスとして `src/Prompts/Profiles/` ディレクトリへ追加し、今後の固有対応もこのディレクトリに集約します。

## 応答整合性

応答は `<item>` 内の `<original>`（原文）と `<translated>`（訳文）を正規表現で読み、**位置ではなく原文内容で**要求項目へ対応付けます。位置（順序）方式は翻訳が1件抜けると以降が全部ずれるため採用しません。内容照合なら、抜けた項目はその1件だけが未解決になり、他は正しく対応します。元のIDへの復元はこちら側のプログラムで行います。照合キーは空白・タブ・改行を空白1つへ畳んで正規化するため（`\s+`→空白1つ＋trim）、モデルが原文エコー時に改行や空白を変形しても一致します。翻訳が得られた項目は確定し、未解決項目だけを再要求し、形式回復後も残る複数項目はバッチを二分して再試行します。

タグ・属性・プレースホルダー・改行などの保持は、送信前のトークン置換（センチネル）ではなく、プロンプトでの明示指示に委ねます。小型モデルは不透明トークンを壊しやすく、かえって復元失敗・再送を招くため、実績のある参照実装と同様にトークン置換は行いません。XMLラッパーの整合のためだけに、送信時に `&`・`<`・`>`・`"`・`'` をエスケープし、応答で復元します。原文エコーが崩れて照合できない項目は誤って他へ割り当てられることはなく、未解決として再試行に回ります。

## APIメッセージ

翻訳規則と、任意のアプリ背景（`AppSummary`）・信頼済み追加指示（`AdditionalInstructions`）はsystemへ入れ、翻訳対象の `<request>` はuserへ入れます。`AppSummary` は用語・語調選択の参考に限る旨を明示し、翻訳の対象にはしません。hy-mt2 専用プロファイルは実績のあるプロンプトをそのまま用いるため、`AppSummary`・`AdditionalInstructions` は注入しません。

`EndpointUrl` はベースURLとして扱います。Ollamaは `/api/chat`、OpenAI互換は `/chat/completions`、Anthropic互換は `/v1/messages` をバックエンド実装側で追加します。初期値は順に `http://localhost:11434`、`https://api.openai.com/v1`、`https://api.anthropic.com` です。

## APIクライアント

対象の `net6.0` 環境で利用できる公式クライアントライブラリがないため、3バックエンドとも `HttpClient` で現行HTTP APIを直接実装します。配布DLLにNuGet実行依存はありません。

## 障害処理

実HTTP並列数はセマフォで制御します。一時エラーだけを再試行します。HTTP要求、応答本文の受信、キュー待ち、操作全体のいずれにもプラグイン独自の時間制限を設けません。DNS、TCP接続、TLS確立などが失敗した場合はOSと.NET HTTPスタックが返す通信エラーとして処理します。接続確立後にローカルLLMの生成が数分以上続いても、時間だけを理由に中断しません。

一時エラーは、HTTP 408、409、425、429、529、5xx、およびHTTP応答前の通信失敗です。最終失敗時はXUnityへ失敗を通知せず原文を返し、連続失敗によるエンドポイント停止を避けます。

## 内部固定値

次の値は公開設定ではなく、エンドポイントの制御仕様です。

| 項目 | 固定値 | 用途 |
|---|---:|---|
| XUnity同時Translate上限 | `50` | `ITranslateEndpoint.MaxConcurrency`。内部バッファへ同時に保留できるXUnity呼び出し数を制限する。 |
| XUnity 1呼び出し当たり件数 | `1` | `MaxTranslationsPerRequest`。XUnity側のバッチに依存せず、単発入力を内部バッファで束ねる。 |
| 前後文脈 | 前1件、後1件 | 最も近い文脈だけを完全な文字列のまま渡す。文字数で途中切断しない。 |
| 形式回復試行 | `2` | 初回プロンプトと回復用プロンプトを各項目へ適用する。その後、未解決の複数件を二分する。 |
| プロバイダ包絡余白 | `128` token相当 | HTTPプロバイダ側の追加表現を見込む保守的な内部推定余白。 |
| 最小出力余白 | `128` token相当 | 短い翻訳でも応答を収めるための内部推定下限。 |
| 出力包絡余白 | `32` token相当 | 出力の区切りや装飾を見込む内部推定余白。 |
| 再試行基準待機 | `500` ms | 一時エラー後の指数バックオフ初期値。0～250msのjitterを加える。 |
| 再試行バックオフ上限 | `30000` ms | 自前の指数バックオフだけに適用する。サーバーの `Retry-After` が長い場合はそちらを優先する。 |
| dispatcher異常時待機 | `100` ms | 予期しない内部例外が連続した場合のbusy loopを防ぐ。 |
| Anthropic API version | `2023-06-01` | Messages APIの必須 `anthropic-version` ヘッダー。 |
| Anthropic `max_tokens` | 動的 | 入力文字列のUTF-8バイト上限推定と出力余白から要求ごとに算出し、最低1とする。固定1024は使用しない。 |
| HTTP応答待ち | 無制限 | `HttpClient`、キャンセルトークン、操作全体の全層で時間制限を追加しない。 |

## ログと秘密情報

`LogLevel` の既定値はInfoです。Infoは初期化完了などの通常動作情報に使います。再試行待機と各翻訳項目の完了はDebug、回復可能な異常はWarn、処理失敗はErrorです。

`LogBatchActivity` は、request ID、件数、文字数、推定トークン数、所要時間を含むバッチ依頼・応答のInfoログを制御します。

出力先は実行ホストの標準コンソールです。`LogFile` の初期値は空であり、既定ではログファイルを作成しません。ファイルパスを指定した場合はコンソールとファイルへ同じ内容を出力し、両方に共通の `LogLevel` を適用します。APIキー、完全なプロンプト、完全な応答は出力しません。例外メッセージ内にAPIキーが含まれた場合も置換します。
